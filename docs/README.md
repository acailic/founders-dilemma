# Documentation

Welcome to the Founder's Dilemma documentation! This directory contains all project documentation organized by purpose.

## Directory Structure

### `/design`

Game design and implementation planning documents.

- **GAME_DESIGN.md** - Complete game design specification
  - Core mechanics and systems
  - Win/defeat conditions
  - Educational philosophy
  - Action catalog and synergies

- **IMPLEMENTATION_PLAN.md** - Development roadmap and technical architecture
  - Feature breakdown
  - Technical approach
  - Implementation timeline

### `/development`

Development process and progress tracking.

- **PROGRESS.md** - Implementation progress tracker
  - Completed features
  - Current status
  - Future work

### `/features`

Feature-specific documentation.

- **github-pages-checker.md** - GitHub Pages health monitoring feature
  - Real-time status monitoring
  - Implementation details

### `/archive`

Historical documents and deprecated content.

- **OFFICE_CAM_TESTING.md** - Testing documentation for office visualization
- **OFFICE_VISUALIZATION_COMPLETE.md** - Completion notes
- **OFFICE_VISUALIZATION_DESIGN.md** - Original design specs
- **SAMPLE_README.md** - Previous README template

## Main Documentation

- **README.md** - Main project documentation (in root)
- **CONTRIBUTING.md** - Contributor guide (in root)
- **CODE_OF_CONDUCT.md** - Community guidelines (in root)
- **SECURITY.md** - Security policy (in root)

## Contributing to Documentation

When adding new documentation:

1. **Place in appropriate directory**
   - Design docs → `/design`
   - Development tracking → `/development`
   - Feature docs → `/features`
   - Outdated/historical → `/archive`

2. **Follow naming conventions**
   - Use UPPER_CASE for major docs (DESIGN.md)
   - Use lowercase-with-hyphens for feature docs (github-pages-checker.md)
   - Be descriptive and specific

3. **Update this README**
   - Add entry in appropriate section
   - Describe the document's purpose

4. **Keep it current**
   - Update docs when features change
   - Move outdated docs to `/archive`
   - Reference related documents

## Documentation Standards

- Use Markdown formatting
- Include clear headings and structure
- Add code examples where helpful
- Keep technical accuracy high
- Link to related documentation
- Update CHANGELOG.md for significant doc changes
